# AI 驱动的 UI 测试工具调研：Midscene 与 Browser-Use

> 🚧 调研进行中

## 为什么关注 AI 测试工具

传统 UI 自动化的痛点：
- 元素定位脆弱（XPath/CSS Selector 随 UI 变更失效）
- 维护成本随用例规模指数增长
- 难以理解页面语义（断言只能比对文本/属性）

AI 带来的变革：
- **视觉定位**：像人一样"看"页面，不依赖 DOM 结构
- **自然语言驱动**：用中文/英文描述操作意图
- **语义断言**：理解页面含义而非死板比对

## Midscene.js

### 原理
- 基于多模态 LLM（支持 GPT-4V / Qwen-VL）
- 截图 → 视觉理解 → 定位元素坐标 → 执行操作
- 支持 Playwright / Puppeteer 集成

### 核心 API

```javascript
// 自然语言操作
await ai('点击登录按钮');
await ai('在搜索框输入"测试"');

// 语义断言
await ai.assert('页面显示"欢迎回来"');
await ai.query('列表中有几条数据？'); // 返回结构化结果
```

### 优势
- 不依赖 DOM 选择器，对 UI 重构容忍度高
- 断言更接近人类测试思维
- 支持可视化调试

### 局限
- 需要 LLM API（成本、延迟）
- 视觉定位有概率性误差
- 复杂交互（拖拽、画布）能力有限

## Browser-Use

### 原理
- Agent 驱动：LLM 作为"大脑"决策每一步操作
- 支持多步骤规划 + 执行 + 验证的闭环
- 开源项目，GitHub 增长迅速

### 与 Midscene 的区别

| 维度 | Midscene | Browser-Use |
|------|----------|-------------|
| 定位方式 | 视觉坐标定位 | 元素索引 + 视觉 |
| 决策模式 | 单步指令执行 | Agent 多步自主规划 |
| 适用场景 | 结构化测试用例 | 探索式操作 / 数据采集 |
| 稳定性 | 较高（单步执行） | 中（多步累积误差） |

## 实践思考

1. **适合什么场景？**
   - 频繁 UI 变更的项目（视觉定位优势明显）
   - 探索式测试（AI Agent 自主发现路径）
   - 不想维护 XPath/CSS Selector 的团队

2. **不适合什么场景？**
   - 需要精确像素级验证
   - 响应时间敏感的压测场景
   - 离线环境（需要 LLM API）
